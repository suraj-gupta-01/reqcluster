# ReqCluster — Metrics, Accuracy & Performance Analysis

All numbers below are **measured** (no estimates unless explicitly marked) on:

- **Hardware:** 12-core CPU laptop + NVIDIA RTX 2050 (4 GB).
- **Architecture:** GPU embeddings (SBERT on CUDA) + UMAP/HDBSCAN on CPU +
  **PostgreSQL** (Docker) + **Redis** cache (Docker), backend running locally
  (`run-gpu.sh`). LLM = mock for the Intelligence-stage timings.
- **Datasets:** synthetic UAV Flight-Management-System requirements with a
  ground-truth subsystem per requirement (the `module` column), generated by
  `data/generate_uav_fms.py` (16 distinct subsystems + a few odd/programmatic
  requirements that should surface as noise).

---

## 1. Clustering accuracy / validation

Computed by `GET /api/metrics` (also shown in the UI's Overview → *Validation
metrics*). Accuracy = **cluster purity** vs the ground-truth subsystem; ARI / NMI /
V-measure are standard external clustering scores (0-1, higher is better);
silhouette (-1..1) rates geometric separation.

| Dataset | N | Clusters | True groups | Noise | **Accuracy (purity)** | ARI | NMI | V-measure | Silhouette |
|---|--:|--:|--:|--:|--:|--:|--:|--:|--:|
| **2k**  | 1,998  | 17 | 17 | 0.7% | **98.3%** | 0.956 | 0.977 | 0.977 | 0.825 |
| **10k** | 9,998  | 20 | 17 | 4.6% | **98.0%** | 0.893 | 0.951 | 0.951 | 0.544 |
| **35k** | 34,990 | 22 | 17 | 6.5% | **95.7%** | 0.811 | 0.912 | 0.912 | 0.091 |

**Reading it:**
- **Accuracy stays 95.7-98.3%** — clustering recovers the true subsystems almost
  perfectly. NMI/V-measure ≥ 0.91 everywhere.
- **Noise stays under 7%** at every size (the goal) — those points are genuine
  boundary cases plus the deliberately-odd requirements.
- **Silhouette falls with size** (0.83 → 0.09): at 35k the clusters are correct
  (high purity) but sit closer together in the 2-D projection, so geometric
  separation looks tighter. Purity/ARI are the more meaningful scores here.
- The key enabler was scaling UMAP `n_neighbors` with N (a fixed 15 gave **38%**
  noise at 35k; size-scaled gives **6.5%**).

## 2. Full-pipeline timing (best case)

Per-stage, measured end-to-end through the live app (GPU + Postgres + Redis),
**warm, all CPU cores**. Seconds.

| Size | Embed (GPU) | UMAP | HDBSCAN + label + graph | Persist (Postgres) | **TOTAL** |
|---|--:|--:|--:|--:|--:|
| **2k**  | 0.6 | 7  | 0.5 | 0.3 | **~9 s** |
| **10k** | 3.4 | 26 | 1.5 | 1.0 | **~32 s** |
| **35k** | 14.1 | 75 | 5.5 | 2.7 | **~97 s** |

- **GPU embeddings** are ~7× CPU (35k: 14 s vs ~78 s on CPU); Redis makes re-runs
  embed-free.
- **UMAP dominates** (CPU). At 35k the low-noise setting (`n_neighbors=70`) makes
  UMAP ~75 s — the deliberate cost of clean clusters (it was ~27 s / 38% noise
  before). HDBSCAN, labeling, ANN graph, and the Postgres write stay small.

### Worst case (measured deltas, not estimates)
- **Cold start** (first clustering after a server start): + SBERT load ~4 s
  + numba JIT ~8 s ≈ **+12 s**, one-time.
- **Single CPU core** (no parallelism): UMAP ~**1.6×** slower (measured 10k:
  38.8 s single vs 23 s all-core).

So worst-case totals (cold + single core): **2k ~21 s, 10k ~60 s, 35k ~155 s.**

## 3. Intelligence-stage timing (Enrichment / Refinement / Dependency tree)

There are two LLM providers and they behave very differently:

- **`local`** — the **real on-device model** (qwen2.5:3b via Ollama, the default
  for the GPU setup). Enrichment makes **one LLM call per requirement**.
- **`mock`** — an offline deterministic provider (no model). Instant-ish but it's
  algorithmic text, not a real LLM. Useful for bulk/demos where you don't need
  real generation.

### Local LLM (qwen2.5:3b on the RTX 2050) — measured
- Loaded **100% on GPU** (2.4 GB), confirmed via `ollama ps`.
- **~5.5 s per requirement** (30-req run: 164 s including the one-time model load).
- So enrichment time ≈ **N × ~5.5 s** — great for a **small selection** or a demo,
  but ~3 h for 2k and impractical for 10k/35k (one sequential call each on 4 GB).
- **Refinement / cluster summaries** make only a few LLM calls (per cluster/
  suggestion), so they're fine on the local model.

### Mock provider compute (no real LLM) — measured
| Size | Enrichment | Refinement | Dependency tree |
|---|--:|--:|--:|
| **2k**  | 19.7 s | 19.4 s | 1.4 s |
| **10k** | 108.6 s | ~101 s | 14.7 s |

Refinement's coherence + silhouette scoring were O(N²); they're now **sampled**
(cap 400 / 2000), so they stay bounded at 35k instead of exploding.

**Recommendation:** use **local** to *demonstrate* real on-device enrichment on a
handful of requirements (or for Refinement summaries); use **mock** only when you
need to bulk-process thousands quickly without a real model.

## 4. Real-world reference (non-synthetic)

For comparison, **PROMISE-exp** (969 real industry requirements, NFR-labelled)
clusters at ~10.9% noise — its labels are NFR *types* (a different taxonomy than
topic), so the security cluster lands ~75% pure while broad "Functional" spreads
across topics. Real data is messier than these clean synthetic sets, which is
expected.

## 5. How to reproduce

```bash
# datasets
python data/generate_uav_fms.py 2000     # -> data/uav_fms_2k_requirements.csv
python data/generate_uav_fms.py 10000
python data/generate_uav_fms.py 35000

# stack (GPU + Postgres + Redis + backend + UI)
./run-gpu.sh            # or .\run-gpu.ps1 in PowerShell

# upload a CSV in the UI, run clustering, then Overview shows the validation
# metrics; or GET /api/metrics?session_id=<id>
```
